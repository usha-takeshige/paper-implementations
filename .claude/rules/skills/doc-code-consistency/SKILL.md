# ドキュメント・コード整合性チェック スキル

MarkdownドキュメントとPythonコードの整合性を検証するスキル。
機械学習・数値計算アルゴリズムの実装プロジェクトを対象とし、
「ドキュメントに定義されたアルゴリズム・手順がコードに実際に実装されているか」を確認する。

**重要な前提**：
- Docstring・コメントは「ドキュメントの写し」であることが多く、実装の証拠にならない
- 実際に動作するロジック（演算・制御フロー・データ変換）が存在するかを見る
- `pass` / `raise NotImplementedError` / コメントアウトは「未実装」

---

## チェックの種類

### 1. アルゴリズム実装チェック
`doc/` 配下のMarkdownに記載されたアルゴリズム・数式・処理手順が、
Pythonコードの**実際のロジック**として実装されているか確認する。

### 2. テスト整合性チェック
`doc/` 配下のテスト仕様に記載されたテストケースが、
`test/` 配下のPythonテストコードに実際に存在し、適切に検証されているか確認する。

---

## 実行手順

### Step 1: プロジェクト構造の把握

```bash
# ディレクトリ構造の確認
find . -type f -name "*.md" | grep -v __pycache__ | sort
find . -type f -name "*.py" | grep -v __pycache__ | grep -v "^./test/" | sort
find . -path "./test/*.py" | sort
```

対象ディレクトリ：
- **ドキュメント**: `doc/` 配下の `.md` ファイル
- **実装コード**: `test/` 以外の `.py` ファイル
- **テストコード**: `test/` 配下の `.py` ファイル

### Step 2: ドキュメントの解析

`doc/` 配下の全Markdownを読み込み、以下を抽出する。

#### アルゴリズム設計書から抽出するもの

見出し（`##`, `###`）を起点に以下を特定する：

- **アルゴリズムのステップ**: 番号付きリスト・フローとして記述された処理手順
- **数式・計算式**: インライン数式（`$...$`）やブロック数式（`$$...$$`）で定義された演算
- **クラス・関数の定義**: コードブロック内のシグネチャ（`def`, `class`）
- **パラメータ**: 入力・出力・ハイパーパラメータの仕様
- **収束条件・終了条件**: ループや反復処理の停止基準

#### テスト仕様書から抽出するもの

- **テストケース名・ID**: 見出しや箇条書きで明示されたテスト項目
- **テスト対象のアルゴリズム・関数**
- **入力条件**: テストデータ・初期値・境界条件
- **期待される出力・挙動**: 数値的な期待値・収束・例外など

### Step 3: コードの実装内容の解析

> **重要**: Docstring・コメント・変数名は参考にしない。
> 実際に演算・制御フロー・データ変換を行うコード行だけを根拠とする。

#### 実装コードから確認するもの

```bash
# クラス・関数の定義一覧
grep -rn "^class \|^def \|^    def \|^        def " \
  --include="*.py" \
  $(find . -name "*.py" | grep -v __pycache__ | grep -v "^./test/")
```

各関数について、**本体のロジック**を目視で確認する：

**実装ありと判断する条件**（いずれかを満たす）：
- 数値演算（`np.`, `torch.`, `scipy.`, 算術演算子）を含む処理がある
- データ変換・集計・比較の実際の式がある
- 外部ライブラリの計算関数を正しい引数で呼び出している
- ループ内に実際の状態更新処理がある

**未実装と判断する条件**（いずれかに該当）：
- 関数本体が `pass` のみ
- `raise NotImplementedError` のみ
- Docstringと `return None` のみ（処理なし）
- コメントアウトされた処理しかない
- `...`（Ellipsis）のみ

#### テストコードから確認するもの

```bash
# テスト関数の一覧
grep -rn "^def test_\|^    def test_" test/ --include="*.py"
```

各テスト関数について確認：
- 対象の関数・クラスを実際にインポート・呼び出しているか
- `assert` 文に意味ある検証式があるか（`assert True` や `assert 1==1` でないか）
- `np.testing.assert_allclose` / `pytest.raises` など数値・例外検証があるか

### Step 4: 照合

#### アルゴリズム実装チェックの照合観点

| ドキュメント記載 | コードで確認すること |
|---|---|
| アルゴリズムのステップ（手順N） | 対応する処理ブロックが実際に存在するか |
| 数式（例: `loss = Σ(y - ŷ)²`） | 等価な演算式がコードに存在するか |
| クラス・関数シグネチャ | 定義が存在し、引数が対応しているか |
| ハイパーパラメータ | `__init__` 等で受け取り、実際に使用されているか |
| 収束条件 | ループの終了条件として実装されているか |
| 初期化処理 | 乱数・ゼロ初期化などが実装されているか |

#### テスト整合性チェックの照合観点

| テスト仕様記載 | テストコードで確認すること |
|---|---|
| テストケース名 | `test_` 関数が存在するか |
| テスト対象の関数・クラス | 実際にインポートして呼び出しているか |
| 数値的期待値 | `assert_allclose` 等で数値検証しているか |
| 境界条件・エッジケース | 対応する入力を渡してテストしているか |
| 例外・エラーケース | `pytest.raises` 等で検証しているか |

## 注意事項

### 機械学習特有のケース
- **数式の等価性**: ドキュメントの数式とコードの演算が「数学的に等価」であれば一致とみなす
  - 例: `Σ(x_i - μ)² / N` と `np.var(x)` は同等
- **ライブラリ委譲**: `sklearn` 等のライブラリへの委譲は「実装あり」として扱う。ただし、ドキュメントが「自前実装」を明示している場合は「乖離あり」
- **数値安定化の追加**: ドキュメントにない `+ 1e-8` などは乖離として報告しない（実装上の改善とみなす）
- **WIP・TODO**: ドキュメントに `TODO`, `WIP`, `予定`, `検討中` が付いた項目は別扱いとして報告する

### Docstring・コメントの扱い（厳守）
- Docstringに「実装済みかのような記述」があっても、対応する実処理がなければ「未実装」
- コメントに数式や手順が書かれていても、それは実装の証拠にならない
- 変数名が数式の記号と一致しても、演算が伴わなければ実装とみなさない

### 出力の調整
- 「サマリーだけ」→ サマリー表と推奨アクションのみ
- 「未実装だけ」→ ❌ セクションのみ
- 「詳細に」→ 各項目に実際のコード行を引用して説明
- 「テストだけ」→ テスト整合性チェックのみ実施
- 「アルゴリズムだけ」→ アルゴリズム実装チェックのみ実施
