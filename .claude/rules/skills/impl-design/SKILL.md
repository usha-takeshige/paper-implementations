# Skill: 実装設計（仕様・アーキテクチャ）

## 概要

`/doc/paper_method.md`をもとに、以下を構造化した設計書を出力する。

- 論文のAlgorithm表に対応したパブリックAPIの定義
- SOLIDの原則に基づくクラス設計
- データクラスの定義
- Mermaidクラス図
- 採用したデザインパターンとその理由

コードの生成・実際の実装は**このSkillのスコープ外**。設計の構造化のみに集中する。

## 設計原則

### SOLIDの原則

設計全体でSOLIDの原則を遵守する。各クラスを設計する際に以下を確認する。

| 原則 | 内容 | 設計への適用 |
|------|------|------------|
| **S** 単一責任の原則 | クラスは1つの責務のみを持つ | 各アルゴリズムステップ・各数式群を別クラスに分離する |
| **O** 開放閉鎖の原則 | 拡張に開いており、修正に閉じている | 複数のアルゴリズムから選べるように変更可能な部分をインターフェース化する |
| **L** リスコフの置換原則 | サブクラスは親クラスの代替として使える | 継承を使う場合、親クラスの契約を守る |
| **I** インターフェース分離の原則 | 使わないメソッドへの依存を強制しない | 大きすぎるインターフェースは分割する |
| **D** 依存性逆転の原則 | 具体ではなく抽象に依存する | 内部クラスは抽象（ABC）に依存させる |

### データクラスの積極的な使用

状態・パラメータ・中間計算結果を保持する場合は、`@dataclass` または `@dataclass(frozen=True)` を積極的に用いる。

- **ミュータブルなデータ**（学習途中の状態など） → `@dataclass`
- **イミュータブルなデータ**（設定値・ハイパーパラメータなど） → `@dataclass(frozen=True)`

データクラスを使うことでクラスの「責務がデータの保持」であることを明示し、振る舞いを持つクラスと明確に区別する。

## パブリックAPI設計の方針

`/doc/paper_method.md`に計算手順（アルゴリズム）が存在する場合、**そのステップをそのままパブリックAPIとして公開する**。

- 論文のステップをそのままパブリックAPIとして公開する
- 各ステップの背後に、責務を分割した内部クラス群を配置する
- 論文のアルゴリズムとコードの対応が自明になるよう、ステップ名称はAlgorithm表に忠実に従う

```python
# 例）論文のAlgorithmが5ステップの場合
class {HandlerName}:
    def {step1_name}(self) -> {ReturnType}: ...
    def {step2_name}(self) -> {ReturnType}: ...
    def {step3_name}(self) -> {ReturnType}: ...
    def {step4_name}(self) -> {ReturnType}: ...
    def {step5_name}(self) -> {ReturnType}: ...
```

## 処理手順

1. **入力ファイルの読み込み**
   - `view` ツールで `/doc/paper_method.md`  を読み込む。
   - `/doc/full_paper.md` は**参照しない**。

2. **Algorithm表の確認**
   - `paper_method.md` のアルゴリズムセクションからステップ数・ステップ名を取得する。
   - これがパブリックAPIのメソッド名の原型になる。

3. **クラス候補の洗い出し**
   - `formula_code_mapping.md` の「クラス構成の概観」を起点に、クラス候補を列挙する。
   - 以下の観点でクラスを追加・分割・統合する。
     - 責務が2つ以上ある → 分割する（単一責任の原則）
     - 変更可能な部分がある → 抽象クラス / Protocolを導入する（開放閉鎖の原則）
     - データのみを保持する → データクラスにする

4. **デザインパターンの検討**
   - アルゴリズムの構造から、適切なデザインパターンを検討する。
   - 採用する場合は採用理由と代替案の却下理由を明記する。
   - よく現れるパターンの候補を以下に示す（強制ではない）。

   | 場面 | 候補パターン |
   |------|------------|
   | 選択肢が複数あるような機能の種類を切り替えたい | Strategy パターン |
   | 複数の機能を動的に組み合わせたい | Composite パターン |
   | 複雑なオブジェクトの構築を分離したい | Builder パターン |
   | アルゴリズム全体のステップの骨格を固定したい | Template Method パターン |
   | ステップ間の状態をオブジェクトとして渡したい | データクラス（Value Object） |

5. **クラス図の作成**
   - Mermaid形式でクラス図を作成する。
   - パブリックAPIのメインクラス・内部クラス群・データクラス・抽象クラスをすべて含める。
   - 依存関係（`-->`）・継承（`<|--`）・集約（`o--`）を明示する。

6. **出力テンプレートに従って整理**
   - 上記フォーマットで出力する。

## 命名規則

- **クラス名・メソッド名**は論文中で使われている語を優先して使う
- **パブリックAPI**のメソッド名は `{論文のステップ名をsnake_case化}` とする（長くならないように）
- **内部クラス**は論文の概念名をそのまま使う（例：`NullSpaceComputer`、`Generator`）
- **データクラス**は保持するデータの概念名を使う（例：`InterpolationSystem`、`TrainingConfig`）

---

## 注意事項

- **論文全文（`/doc/full_paper.md`）は参照しない**：`paper_method.md`のみを使う
- **コードを生成しない**：設計書・クラス図の出力に留める
- **デザインパターンを強制しない**：適切でない場合は「パターン不採用」と明記してよい
- **SOLIDの原則の適用箇所を明示する**：なぜそう設計したかを各クラスのSOLIDチェックに記載する
- **データクラスを使う理由を明示する**：「データの保持が責務であるため」と記載する
